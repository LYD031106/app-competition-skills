# Report Writing Guide

## Goal

默认输出“高信号变化报告”，不是“大而全的数据扫描”。

默认只回答 4 件事：
- 我方这期最值得盯的关键词发生了什么
- 哪些竞品真的改了 `Name / Title / 前三张宣传图`
- 这些变化对我方意味着什么
- 现在该做什么

除非用户明确要求扩展，否则默认范围固定为：
- 目标产品：只跟 `target_keywords` 的真实 rank 变化
- 竞品：只看 `Name / Title / 前三张宣传图` 的实质变化
- 发版、评论、定价、开发者矩阵、similar apps：只在能解释上述变化时才进入正文

当项目开启飞书交付时，仍采用“双层交付”：
- 主摘要文档：1-2 分钟扫读版
- 详细子文档：保留证据、详细表格与附录

不要把“主摘要文档”理解成“删减版完整报告”。
主摘要文档是独立交付物，不能把 `Evidence Appendix`、完整 release 表或全量关键词表硬塞进去。

## Default Structure

推荐顺序：
1. `一句话结论`
2. `目标产品关键词变化`
3. `竞品动态`
4. `建议行动`
5. `Evidence Appendix`
6. `Limitations`

仅在确实有解释价值时，再补：
- `Release Context`
- `Secondary Market Cross-Check`
- `Name Collision / Ambiguity Alert`

如果是飞书主摘要文档，推荐顺序改为：
1. `封面摘要栏`
2. `一句话判断`
3. `目标产品关键词变化`
4. `竞品变化雷达`
5. `现在要做什么`
6. `子文档入口`
7. `范围与局限`

## Core Writing Pattern

每个模块都按这个顺序写：
1. 先写变化
2. 再写对我方的意义
3. 最后决定是否需要动作

没有变化的内容，不要硬写进正文占版面。
如果某个模块只能写“无变化”，优先用 1 句话带过，不展开长表。

## Body Scope Rule

- 正文默认只写目标 app 与本期确有实质变化的 3-5 个竞品。
- 没变化的关键词不要逐条写“无变化”。
- 没变化的竞品不要进入正文。
- 竞品关键词默认不写，除非用户明确要求恢复这条监控。
- 主摘要文档默认最多写 3-5 个对象、最多 5 行关键词变化、最多 3 条动作。
- 详细子文档继续保留详细证据，但入口层仍必须遵守 4 段式结构。

## App Link Rule

- 报告中凡是进入正文、主摘要、详细子文档、主表或附录的 app，默认把首次出现的 `app_name` 写成 Markdown 链接：`[App Name](app_store_url)`。
- 表格里的 `对象`、`目标 App`、`竞品名称`、`leader_app_name` 等 app 名称列，也优先使用可点击名称，而不是另起一列堆 URL。
- 链接优先级：MCP 返回的官方商店链接 > 由 iOS `app_id` 拼出的 App Store 链接 > 由 Android package name 拼出的 Google Play 链接 > 证据文件里的已核验页面链接。
- 首屏仍然不要暴露裸 `app_id` / package name；它们只用于生成链接或放在附录、JSON、证据备注里。
- 如果某个 app 进入报告但无法可靠生成链接，必须在 `Evidence Appendix` 或 `Limitations` 写明“链接缺失 / 待核验”，不要默默省略。

## ASO Explanation Lens

如果需要解释“为什么这次关键词或素材变化值得动作”，默认用以下 ASO lens：
- `Keyword truth`：只基于 `target_keywords` 和实际 rank
- `Store audit`：看 `Name / Title / Top-3 Screenshot` 是否和目标意图对齐
- `CRO`：看首屏表达是否更容易理解，是否可能影响点击和转化

默认不要写方法论术语本身。
把它们改写成业务语言，例如：
- `竞品首屏把主卖点从 X 换成了 Y`
- `当前前三图更快解释了去物场景`
- `评分与近期评论在帮它降低用户犹豫`

## Section Guidance

### 一句话结论

- 只写 1 句，尽量不超过 30 个汉字。
- 必须用大白话说清“本期最值得记住的变化是什么”。
- 优先把“我方关键词位移”与“最重要的竞品变化”压缩到一句话里。

示例：
- `主词 video eraser 从第 7 掉到第 9，新同名竞品冲到第 8。`

### 目标产品关键词变化

- 只列 `target_keywords` 中发生变化的词，最多 5 个。
- 默认表头固定为：
  - `关键词`
  - `上期排名`
  - `本期排名`
  - `变化`
  - `备注`
- `备注` 只写原因、背景或旁证，不写建议。
- 没有历史 rank 时，明确写 `baseline`。
- 如果所有目标关键词都没有显著变化，只写一句：
  - `本期目标关键词未见需要升级处理的位移。`
- 如需解释原因，优先检查：
  - 是否有同名 / 近同名干扰
  - 是否发生 `Name / Title` 变化
  - 是否发生前三图变化
  - 是否能由 release / reviews 补充解释

### 竞品动态

- 只列本期有实质变化的对手，最多 5 个。
- 每个竞品最多 3 行，不要写成小作文。
- 固定回答两件事：
  - `变化`：`Name` 改了 / `Title` 改了 / 前三图换了 / 顺序变了 / 主题变了
  - `对我的影响`：定位更接近了 / 点击吸引力增强了 / 暂时无明显影响 / 需要继续观察
- 不要写竞品关键词变化。
- 如果平台没有独立 `Title` 字段，用 `Subtitle` 或首屏主标题文案代理，并在证据里说明口径。

### 建议行动

- 最多 3 条，按优先级排。
- 每条动作必须包含：
  - `做什么`
  - `为什么`
  - `什么时候前完成`
- 动作要落到“改词、改图、补验证、继续观察”这类可执行事项，不能只写空泛口号。

### Evidence Appendix

- 只保留支撑结论所需的事实、来源和补充说明。
- `app_id`、精确时间戳、工具状态标签放这里，不放首屏。
- 附录里每个涉及 app 的证据行必须保留 `app_store_url` 或说明链接缺失原因，便于回查。

### Limitations

- 明确写出：
  - 历史缺口
  - MCP 覆盖缺口
  - 不能得出的结论
- 如果截图来自网页回退，正文写自然语言，技术标签放附录。

## Tone Rules

- 用研究口吻，不用营销腔。
- 结论明确，但不要装作比证据更确定。
- 不要把报告写成 API 字段转抄。
- 先写变化，再写解释，不要先铺长背景。

## Common Mistakes

- 先写背景，真正结论埋在后面
- 把没变化的关键词和竞品也写进正文
- 把竞品关键词模块继续塞进默认摘要
- 只写“截图变了”，不写“表达重点从什么变成了什么”
- `备注` 里混入建议，导致“变化表”和“行动项”职责不清
- 没有历史时硬写升降趋势
- 把 `MCP 未返回截图` 误写成 `App Store 页面无截图`
- 把主摘要文档写成删减版完整报告，而不是独立摘要交付物

## Completeness Check

在交付月报前，快速确认：
- 有 `comparison_window`
- 有 `latest_collection_at`
- 有 1 句能直接说清重点的结论
- 关键词模块只列目标产品、只列变化词
- 竞品模块只列实质变化对象，且只围绕 `Name / Title / 前三图`
- 动作模块不超过 3 条，且每条都带截止时间
- 有 `limitations`
- 如果开启飞书交付，有独立主摘要文档和详细子文档
- 主摘要文档没有把完整 `Evidence Appendix` 重新塞回首屏
