# Executive Main Doc Contract

## Goal

主摘要文档是管理者快速扫读入口，不是删减版完整报告。

默认目标：
- 30 秒看到结论
- 90 秒抓住重点
- 3-5 分钟完成决策性扫读

## Fixed Structure

主摘要文档默认固定为：
1. 判断型标题
2. `封面摘要栏`
3. `一句话判断`
4. `目标产品关键词表现`
5. `竞品变化雷达`
6. `现在要做什么`
7. `子文档入口`
8. `范围与局限`

## Required Content

必须进入主摘要文档的内容：
- 本期一句话判断
- 目标产品关键词表现
- 竞品 `Name / Title / 前三图` 变化
- `目标 App` 与进入摘要的竞品名称必须优先写成可点击商店链接
- 每个进入摘要竞品的当前 App Store `top1-3` 宣传图
- 1-3 条动作建议
- 详细子文档入口

必须下沉到详细子文档的内容：
- 完整 `Evidence Appendix`
- 完整 `Release Context` 行表
- 完整 `Top-3 Screenshot` 逐对象语义分析
- `top1-3` 逐张原图与逐张证据说明
- 全量关键词 rank 表
- 方法论和原始字段解释

## First-Screen Rules

首屏必须满足：
- 标题表达判断，不只写主题
- 结论先于元数据
- 动作块只保留 1-3 条最该做的事
- 术语全部改写成业务中文
- 关键词表只保留 `target_keywords` 中的变化词
- 竞品表只保留高信号对象
- 图组只保留进入摘要的竞品当前 `top1-3`

首屏禁止出现：
- `app_id`
- 裸 URL
- 原始状态码
- 工具内部标签
- 精确采集时间戳
- 完整证据长表

## Action Dedupe Rule

- 同一条核心动作只能有一个主表达位
- `一句话判断`、`现在要做什么` 之间不能重复堆叠同一句动作
- 如果需要重复提及，只能补充上下文，不重复原句

## Business Language Rule

摘要层所有术语必须业务化：
- `tool_empty_web_present` -> “工具未直接抓到截图，但网页确认商店页有图”
- `tool_empty_web_empty` -> “工具和网页检查都没有发现商店页截图”
- `inference` -> “基于现有证据的保守判断”

原始标签保留到详细子文档、附录或 JSON。

## Summary Tables

主摘要文档必须至少有两张窄表：

### 目标产品关键词表现表

列固定为：
- `关键词`
- `上期排名`
- `本期排名`
- `变化`
- `备注`

关键词分组默认写成：
- `At-Risk Target Keywords`
- `Recoverable Target Keywords`

### 竞品变化雷达表

列固定为：
- `对象`
- `Name 变化`
- `Title 变化`
- `前三图变化`
- `对我的影响`

`对象` 列使用 `[App Name](app_store_url)`，不要另起 URL 长列。

## Summary Image Block

主摘要文档必须在 `竞品变化雷达` 段内带上当前宣传图图卡：
- 只覆盖进入摘要版 `竞品变化雷达` 的竞品
- 每个竞品插入 1 张基于当前 App Store `top1-3` 生成的三联合并图
- 三联合并图必须保留 `top1 -> top2 -> top3` 顺序，不能退化成随意挑选的单张代表图
- 合并图至少带 1 行 caption，写清 `对象 + top1-3 + market + collected_at`
- 如果来自网页回退，明确写“网页回退取回”
- 如果图缺失，显式写缺口说明而不是静默跳过
- 默认把合并图紧跟在 `竞品变化雷达` 表后，按竞品逐组展开，视为同一段交付，不要再拆成独立的“当前前三图摘记”章节
- 如果用户口头说“放到那个表格里面”，优先理解为“图要和该竞品表格事实紧邻展示”，而不是机械地把多张原图塞进 `lark-table` 单元格
- 使用 `lark-cli docs +update` 写飞书时，默认先在本地生成三联合并图，再插入飞书；不要把多张 `<image>` 直接塞进 `lark-table` 单元格
- 只有在你有更底层的块编辑方案、并且已验证不会破坏表格结构时，才允许真的把图塞进单元格

## Budget

- 发现条数：3-5 条
- 动作条数：1-3 条
- 对象数量：3-5 个
- 关键词摘要：最多 5 行
- 禁止完整 evidence 表
- 禁止大段 maintenance 更新流水账
- 禁止把广义 opportunities pool 写进主摘要

## Anomaly-First Rule

- 优先写“有变化且有意义”的对象
- maintenance 更新只有在改变判断时才进入主摘要
- 近似标题压力、高意图词位移、前三图表达重心变化优先级更高

## Feishu Rendering

优先使用：
- 封面表格
- 1 个结论 callout
- 1 张目标产品关键词表现表
- 1 张竞品变化表
- 1 组放在 `竞品变化雷达` 段内、紧跟表格并按竞品展开的当前 `top1-3` 宣传图
- 1 个动作 callout
- 详细子文档链接

不要在主摘要文档里重复放完整正文目录。
不要把“竞品图组”另起成会让读者误以为脱离 `竞品变化雷达` 的独立章节。
