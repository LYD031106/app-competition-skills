# Report Rendering Style

## Audience

默认读者是产品 / ASO 负责人和管理层。写法必须支持：
- 30 秒看到结论
- 90 秒抓住重点
- 3-5 分钟完成决策性扫读

如果开启飞书交付，默认分为两层：
- 主摘要文档：管理者快速扫读
- 详细子文档：保留完整长报告结构，但前部先做决策入口层

## Name-First Rule

- 正文默认写可点击的 `app_name`，格式为 `[app_name](app_store_url)`
- 主表格默认只显示可点击名称，不显示 `app_id`
- 首次出现可用“完整名称 + 角色短标签”
- 名称非常接近时，优先用“名称 + 角色 / 开发者 / 定位短标签”区分
- `app_id` 只保留在附录、JSON、证据备注和必要的歧义补充里

链接来源优先级：
1. MCP 返回的官方商店链接，例如 Apple `trackViewUrl` 或 Google Play `url`
2. iOS `app_id` 拼接出的 `https://apps.apple.com/app/id<app_id>`
3. Android package name 拼接出的 `https://play.google.com/store/apps/details?id=<package_name>`
4. 已核验的证据页面链接

如果链接无法可靠确认，正文仍保留名称，但必须在附录或 `Limitations` 标记“链接缺失 / 待核验”。

## Ambiguity Rule

- 如果 `target_app` 或正文重点对象存在同名 / 近同名 app，首屏必须在 `一句话结论` 后补一个小型歧义消解表
- 该表至少写清：`app_name`、角色短标签、开发者 / 公司短标签、为什么容易混淆
- `app_id` 下沉到附录或证据备注，不在首屏直接出现
- 不能只靠单复数差异让读者自己猜，也不能只把区分信息放进附录

## Structure Rule

报告优先级高于元数据：
- 结论在前
- 目标产品关键词变化在前
- 竞品动态在前
- 动作在前
- 证据附录在后
- 次市场后置

详细子文档默认分两层：
- `决策入口层`
- `研究证据层`

主摘要文档优先级更高于详细报告：
- 先给一句话判断
- 再给目标产品关键词变化
- 再给竞品变化表
- 再给每个重点竞品 1 张基于当前 `top1-3` 生成的三联合并图
- 再给动作
- 最后才给详细子文档入口

## First-Screen Rule

首屏允许：
- 判断型标题
- 1 句结论
- 目标产品关键词变化
- 竞品实质变化
- 1-3 条动作
- 必要的局限提醒

首屏禁止：
- `app_id`
- 裸 URL
- 原始状态码
- 工具内部标签
- 精确采集时间戳
- 完整 evidence 长表
- 大段 maintenance 流水账

## Table Rule

窄表优先：
- 表内放事实
- 表后放推断
- 单元格避免整段话
- 列名尽量中文化、业务化
- 主摘要表只保留高信号对象或变化词

## Tone Rule

- 用研究口吻，不用 API 转抄口吻
- 不要把工程字段堆在首屏
- 不要让 `report_type`、`comparison_window`、`baseline_ref` 抢首屏注意力
- 摘要层不要直接暴露 `tool_empty_web_present` 这类工具术语

## Action Rule

- 同一条核心动作只能有一个主表达位
- `一句话判断`、`现在要做什么`、详细层动作之间不能重复堆叠同一句动作
- 其他段落只补充上下文

## Limitation Rule

- 局限不只放文末
- 如果某个模块依赖推断或缺历史资产，在模块旁即时提示
- 如果截图来自网页回退而不是 MCP，要在模块旁即时提示“工具抓取存在覆盖缺口”
- 摘要层用自然语言描述局限，不直接贴技术标签

## Good First Screen

详细子文档首屏通常只需要：
- 1 个判断型标题
- 1 个 `一句话结论`
- 1 个歧义消解小表（仅在存在同名 / 近同名风险时）
- 1 张目标产品关键词变化表
- 1 个竞品动态块
- 1 个动作块

主摘要文档首屏通常只需要：
- 1 个判断型标题
- 1 个封面摘要栏
- 1 个一句话判断
- 1 张目标产品关键词变化表
- 1 张竞品变化表
- 1 个“现在要做什么”动作块
- 1 个详细子文档链接区
